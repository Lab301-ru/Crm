> Статус: **на утверждении**.
> Проверено: автотест ролевой матрицы на живом PostgreSQL 16 — **14/14 проверок OK** (`scripts/test-db.sh`); `vite build` + `tsc` strict — чисто.

## 1. Трёхслойная модель защиты

1. **RLS (строки):** какие строки видит/меняет роль — политики Этапа 5.
2. **Колоночные триггеры (поля)** — *новое на этом этапе*: RLS позволяла мастеру обновлять строку своего заказа целиком, включая предоплату и даже `client_id`. Закрыто миграцией `20260609100007_role_guards.sql`:
   - системные поля заказа (`client_id`, `device_id`, `number`, `display_number`, `qr_token`) неизменяемы ни для кого, кроме админа;
   - мастеру запрещены финансы, назначения, гарантия, заявленная неисправность — доступны диагностика, комментарии, срок готовности, статусы и строки работ;
   - мастер не меняет категорию/бренд/модель устройства (сломал бы контекст валидации доп-полей).
3. **UI (видимость):** недоступные действия скрыты, а не задизейблены — интерфейс мастера проще, как и его работа.

## 2. Матрица «роль × действие» (как реализовано и проверено)

| Действие | Admin | Manager | Master |
|---|---|---|---|
| Видеть заказы | все | все | **только свои** |
| Создать заказ / клиента | ✓ | ✓ | — |
| Смена статуса | любые переходы | по таблице переходов | по таблице, только свои |
| Диагностика, комментарии, срок, работы/запчасти | ✓ | ✓ | свои заказы |
| Финансы (предоплата, оплата) | ✓ | ✓ | — (поля скрыты + триггер) |
| Назначение мастера | ✓ | ✓ | — |
| Клиенты: просмотр / правка | все / ✓ | все / ✓ | только своих / — |
| Выручка на дашборде | ✓ | ✓ | — (сервер отдаёт null) |
| Задачи «позвонить» | ✓ | ✓ | — (RLS + скрыто) |
| Справочник: чтение / правка / импорт | ✓/✓/✓ | ✓/быстрое добавление/— | ✓/—/— |
| Шаблоны полей, статусы, правила уведомлений, организация | ✓ | чтение | чтение |
| Сотрудники (создание, роли, пароли) | ✓ | — | — |
| Аудит | ✓ | — | — |
| Soft delete (заказы, клиенты, строки, файлы) | ✓ | ✓ | — |
| Аноним | — | — | — (ни одной строки ни в одной таблице) |
| Деактивированный сотрудник | отрезан от данных немедленно (RLS), вход закрыт (ban) | | |

## 4. Что именно проверяет автотест (все — OK)

Менеджер: видит все 3 заказа · не читает аудит · не правит org_settings (0 строк) · не вставляет статусы (42501).
Мастер: видит только свой заказ (и в `order_list`) · видит только своего клиента · сохраняет диагностику · **предоплата отклонена триггером** · `create_order` отклонён · чужой статус отклонён · свой статус по цепочке работает · outbox пуст и выручка null.
Аноним: ноль строк в orders и clients.
Деактивированный мастер: ноль строк немедленно, без ожидания истечения JWT.

## 5. Риски и открытые вопросы

- UI-скрытие — это удобство, не защита: каждый «скрытый» путь продублирован серверным запретом (см. матрицу — у каждой ячейки «—» есть RLS-политика, триггер или проверка в RPC).
- Тест запускается одной командой и войдёт в CI на Этапе 16.
