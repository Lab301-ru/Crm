# ЭТАП 10: Хранилище файлов (Storage + сжатие фото)

> Статус: **на утверждении**.
> Проверено: `vite build` + `tsc` strict — чисто; прогон всех миграций + RLS-тест (14/14 OK) — миграция Storage корректно пропускается на чистом PostgreSQL и применяется на Supabase.

---

## 1. Что реализовано

**Суть:** фото устройства снимаются на телефон при приёмке и диагностике, сжимаются на клиенте (canvas, длинная сторона ≤1920px, JPEG 75%), грузятся в приватный бакет Supabase Storage и показываются галереей в карточке заказа. Доступ — строго через видимость заказа (RLS), показ — через signed URL на час.

| Компонент | Решение |
|---|---|
| **Бакеты** | `device-photos` и `documents`, оба **приватные** (никаких public URL — фото устройств и акты не для посторонних) |
| **Привязка к заказу** | путь файла `<order_id>/<uuid>.jpg` — первый сегмент пути связывает объект Storage с заказом |
| **RLS Storage** | подзапрос к `orders` внутри политики сам применяет RLS заказов: admin/manager видят всё, **мастер — только фото своих заказов** |
| **Загрузка** | любой активный сотрудник с доступом к заказу (мастер фотографирует диагностику) |
| **Удаление** | soft delete в `attachments` (`deleted_at`/`deleted_by`) — менеджер/админ; у мастера кнопки нет, RLS на `attachments` дублирует запрет; физический DELETE объекта — менеджер/админ (обслуживание) |
| **Сжатие** | canvas: кадр телефона 3–8 МБ → ~150–300 КБ; не картинка или выигрыша нет — грузится оригинал |
| **Метаданные** | таблица `attachments` (Этап 3): kind, путь, имя, mime, размер, кто загрузил; при ошибке вставки строки загруженный файл удаляется из бакета (нет сирот) |

## 3. Матрица доступа к файлам (3 слоя, как на Этапе 9)

| Действие | Admin | Manager | Master |
|---|---|---|---|
| Видеть фото заказа | все | все | только своих заказов (RLS Storage + RLS attachments) |
| Загрузить фото | ✓ | ✓ | в свои заказы |
| Удалить фото (soft) | ✓ | ✓ | — (кнопка скрыта + RLS attachments_update) |
| Физически удалить объект | ✓ | ✓ | — (политика Storage) |
| Документы (PDF) | чтение через заказ; пишет только Edge Function (service_role) | | |
| Аноним | — ни одного объекта (бакеты приватные, политик для anon нет) | | |

## 4. Free tier

1 ГБ Storage: ~300 заказов/мес × 2–3 фото × ~250 КБ ≈ **180–270 МБ/год** — зелёная зона на годы. PDF-документы не хранятся вечно (Этап 11: снимок данных в `order_documents.snapshot`, файл воспроизводим).

## 5. Деплой

Бакеты создаются самой миграцией (`insert into storage.buckets … on conflict do nothing`) — отдельных действий в консоли Supabase не требуется.

## 6. Риски и открытые вопросы

- Signed URL живут час; React Query обновляет их через 30 минут (`staleTime`) — устаревшая подпись в открытой вкладке практически исключена.
- HEIC с iPhone: Safari сам декодирует HEIC в canvas → на выходе обычный JPEG; в остальных браузерах файл уйдёт оригиналом (редкий случай, объём терпимый).
- Политики Storage нельзя проверить в локальном RLS-тесте (нет схемы storage) — компенсировано тем, что они дословно повторяют проверенный паттерн `attachments_select` («подзапрос к orders сам применяет RLS»), плюс ручная проверка на живом Supabase при деплое (Этап 17).
