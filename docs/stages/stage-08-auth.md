## 1. Схема аутентификации

- **Supabase Auth, email + пароль.** Self-signup отключён (включается в Dashboard: Auth → Providers → Email → «Disable new user signups»): сотрудников создаёт только администратор.
- **Сессии:** JWT (1 час) + refresh token; supabase-js продлевает токен автоматически, `onAuthStateChange` в `AuthProvider` ловит вход/выход/обновление. Протухшая сессия → автоматический redirect на `/login` через Protected-гард.
- **Роль в JWT:** `app_metadata.role` — выставляется только через Admin API (service_role), клиент изменить не может. RLS-политики читают её без обращения к таблицам (`app_role()`), `profiles.role` — зеркало для UI.

## 3. Ключевые решения

- **Создание сотрудников — из UI приложения,** а не из Supabase Dashboard: админ нажимает «+ Сотрудник», Edge Function через Admin API создаёт auth-пользователя с `app_metadata.role` и строку `profiles` атомарно (при сбое профиля auth-пользователь удаляется — не остаётся «осиротевших» логинов).
- **Смена роли синхронизирует оба места** (JWT-метаданные + таблица) в одном вызове. Важно: сотрудник увидит новую роль после обновления токена (≤1 час) или перелогина — это поведение JWT, задокументировано.
- **Деактивация = двойное отсечение:** `ban` в Auth (вход невозможен) + `profiles.is_active=false` (действующая сессия теряет доступ к данным через RLS немедленно, без ожидания истечения JWT). Самого себя деактивировать/разжаловать нельзя — защита от «выстрела в ногу» последним админом.
- **Восстановление пароля — двумя путями:** самостоятельно через письмо («Забыли пароль?» → `/reset-password`) и администратором (кнопка «Пароль» в списке сотрудников, новый пароль сообщается лично).

## 4. Первичная настройка (bootstrap первого администратора)

Выполняется один раз при развёртывании (войдёт в инструкцию Этапа 17):

1. Supabase Dashboard → Authentication → Add user → email + пароль (`Auto Confirm` ✓).
2. SQL Editor:

```sql
-- роль в JWT
update auth.users
set raw_app_meta_data = coalesce(raw_app_meta_data, '{}'::jsonb) || '{"role":"admin"}'::jsonb
where email = 'admin@example.com';

-- профиль
insert into public.profiles (id, full_name, role)
select id, 'Администратор', 'admin' from auth.users where email = 'admin@example.com';
```

3. Auth → Providers → Email → отключить self-signup.
4. Auth → URL Configuration → Site URL = адрес SPA (для ссылок восстановления пароля).

Дальше все сотрудники создаются из приложения.

## 5. Риски и открытые вопросы

- Окно «старая роль в JWT до 1 часа» после смены роли: понижение прав мгновенно отрезается деактивацией+активацией (сбрасывает сессии), для штатных случаев приемлемо.
- Rate limit писем восстановления — стандартный supabase (защита от спама), отдельной настройки не требует.
