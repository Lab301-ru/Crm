# ЭТАП 13: PWA (установка на телефон, офлайн)

> Статус: **на утверждении**.
> Проверено: `vite build` + `tsc` strict — чисто; service worker собран (precache 18 файлов, ~1.2 МБ), манифест и registerSW в сборке.

---

## 1. Что даёт PWA сотруднику СЦ

- **Установка на главный экран** телефона/планшета: полноэкранный запуск без адресной строки (`display: standalone`), тёмная иконка с гаечным ключом, заставка в цветах темы.
- **Открывается без сети**: оболочка приложения в precache service worker — мастер в подвале без Wi-Fi открывает CRM и видит свои заказы.
- **Читается без сети**: кеш React Query персистится в localStorage на сутки — списки, карточки заказов и справочники, открытые ранее, доступны офлайн.
- **Работает без сети**: смена статуса и сохранение диагностики встают в **офлайн-очередь** и доотправляются при появлении сети — даже после перезагрузки страницы.
- **Жёлтый баннер** «Нет сети — показаны сохранённые данные…» объясняет состояние, чтобы зависшая кнопка «Сохранить» не пугала.

## 2. Архитектура офлайна: два слоя с разной ответственностью

| Слой | Что кеширует | Почему так |
|---|---|---|
| **Service worker (Workbox, generateSW)** | только статику: JS/CSS/HTML/иконки + SPA-fallback на index.html | данные Supabase SW не трогает — у SW-кеша нет понятия ролей и инвалидации |
| **React Query persist (localStorage, ключ `crm-cache`)** | данные запросов (gcTime = сутки) и очередь мутаций | инвалидация, роли и retry уже живут здесь; восстановление кеша → `resumePausedMutations()` → `invalidateQueries()` |

Очередь переживает перезагрузку только для мутаций с зарегистрированными defaults (`main.tsx`): **смена статуса** и **обновление заказа** (диагностика/комментарии/оплата) — главные действия мастера «в поле». Прочие мутации при офлайне тоже встают на паузу, но требуют открытой страницы.

**Безопасность:** при выходе из аккаунта офлайн-кеш стирается (`queryClient.clear()` + удаление `crm-cache`) — данные CRM не переживают сессию на общем устройстве.

## 4. Ограничения (осознанные)

- **Создание заказа офлайн не поддерживается**: приёмка происходит у стойки с сетью, а офлайн-создание тянет за собой дедупликацию клиентов и конфликt номеров — несоразмерная MVP сложность (Phase 2, если понадобится).
- Фото офлайн не грузятся и не кешируются (тяжёлые, очередь блобов в localStorage не живёт) — загрузка просто упадёт с понятной ошибкой, можно повторить при сети.
- localStorage ~5 МБ: кеш списков и карточек укладывается с большим запасом; при переполнении персист молча отключается — приложение продолжает работать онлайн.
- iOS: установка только через Safari → «Поделиться» → «На экран Домой» (баннер установки Apple не показывает) — попадёт в инструкцию Этапа 17.
