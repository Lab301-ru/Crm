## 1. Архитектурное решение: печатная форма вместо серверного PDF

В плане Этапа 6 значилась генерация PDF в Edge Function. Реализовано иначе — **HTML-формы для печати в браузере**, и вот почему:

| Критерий | Edge Function + pdf-lib | Печатная форма (выбрано) |
|---|---|---|
| Кириллица | нужен TTF-шрифт в функции (~700 КБ, хостинг, лицензии) | нативные шрифты браузера — идеально из коробки |
| Free tier | расход Edge-инвокаций + хранение PDF (1 ГБ) | ноль: рендер на клиенте, файл не хранится |
| «Сохранить как PDF» | да | да — встроено в диалог печати любого браузера, включая телефон |
| Скорость у стойки | запрос → генерация → скачивание | мгновенно: открыл вкладку — диалог печати уже на экране |
| Как в Workpan | — | да, Workpan печатает формы из браузера |

**Воспроизводимость** при этом не потеряна: документ печатается **только из снимка** в `order_documents.snapshot` (jsonb). Квитанция, распечатанная повторно через месяц, идентична первой — даже если заказ, клиент или цены изменились. Колонка `storage_path` осталась (стала nullable, миграция 9) — задел для Phase 2 (отправка PDF клиенту по email).

## 2. Четыре документа

| Документ | Когда доступен | Содержимое |
|---|---|---|
| **Квитанция о приёме** | всегда | реквизиты СЦ, клиент, устройство (включая доп-поля категории), заявленная неисправность, предоплата, срок, дисклеймер из настроек, **QR-код отслеживания**, подписи |
| **Акт выполненных работ** | есть работы/запчасти | диагностика, таблица позиций с итогами, предоплата, к оплате, подписи исполнителя (мастер) и заказчика |
| **Акт выдачи** | статус «готов» / «выдан» | устройство, позиции, оплата (статус и способ), гарантия до даты, «претензий не имею», подписи |
| **Гарантийный талон** | задан срок гарантии | работы под гарантией, срок и дата окончания, условия гарантии, подпись и печать |

## 3. Как устроено

- Маршрут `/orders/:id/print/:docType` — вне Layout (на бумаге не место сайдбару), светлая «бумажная» тема A4, автозапуск `window.print()` когда форма и QR готовы.
- Логика снимка: открыли печать → берётся последний снимок этого типа; снимка нет → создаётся из живых данных. Ссылка `?refresh=1` («печать со свежими данными») всегда создаёт новый снимок; история снимков накапливается в `order_documents`.
- Снимок самодостаточен: реквизиты организации, клиент, устройство с **разрешёнными** доп-полями (label+value на момент печати — переименование шаблона поля не меняет старые документы), позиции, суммы, имена менеджера/мастера.
- Карточка «Документы» в заказе: четыре кнопки с подсказками о доступности и датой созданного снимка. **Скрыта у мастера** (печать — задача приёмки/выдачи; вставка снимков и так запрещена RLS `documents_insert -> is_manager_up`, страница печати редиректит мастера на заказ).
- QR-код — пакет `qrcode`, грузится динамическим импортом только на странице печати (~10 КБ gzip отдельным чанком).

## 5. Риски и открытые вопросы

- Сумма прописью в актах не печатается (для ИП/самозанятых не обязательна) — кандидат в Phase 2 вместе с email-отправкой PDF.
- Дата отсчёта гарантии = дата создания снимка талона/акта выдачи (печатают при выдаче). Если талон напечатать заранее, дата уйдёт — лечится перепечаткой `?refresh=1` в день выдачи.
- Шапка использует реквизиты из настроек организации; пока они не заполнены, печатается только название — заполнить на Этапе 17 (деплой).
