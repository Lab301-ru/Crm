# ЭТАП 1: Анализ требований

> CRM для сервисного центра по ремонту бытовой техники, электроники и цифровых устройств.
> Статус: **на утверждении**. Следующий этап (архитектура) не начинается до явного подтверждения.

---

## 1. Разбор постановки

### 1.1. Что строим

Production-ready CRM уровня Workpan/RemOnline для одного сервисного центра (1–10 сотрудников), mobile-first, на русском языке. Ядро системы — **заказ на ремонт**, вокруг которого живут клиенты, устройства, финансы, документы, уведомления и аудит.

### 1.2. Доменная модель (как мы её понимаем)

```
Клиент 1 ──── * Заказ * ──── 1 Устройство
                 │
                 ├── Неисправность (заявленная → диагностика → работы)
                 ├── Финансы (работы + запчасти − предоплата = итог)
                 ├── История статусов (state machine, immutable)
                 ├── Файлы (фото, документы)
                 ├── Документы PDF (квитанция, акты, гарантийный талон)
                 ├── Уведомления (Telegram / WhatsApp / Email)
                 └── QR → публичная страница статуса (по токену)

Справочник: Категория → Бренд → Модель (динамический, импорт Excel/CSV)
Категория → шаблоны доп-полей → JSONB-значения у устройства
Роли: Администратор / Менеджер / Мастер (RBAC + RLS)
```

### 1.3. State machine статусов

Основной поток:

```
Не принят → Принят → Диагностика → Ожидание согласования
  → Ожидание запчастей → В ремонте → Готов → Выдан
```

Терминальные ветки: `Отказ`, `Утиль`.

Предлагаемые правила переходов (фиксируем как допущение, см. §3):

| Из | Разрешённые переходы |
|---|---|
| Не принят | Принят, Отказ |
| Принят | Диагностика, Отказ |
| Диагностика | Ожидание согласования, В ремонте, Готов*, Отказ |
| Ожидание согласования | Ожидание запчастей, В ремонте, Отказ |
| Ожидание запчастей | В ремонте, Ожидание согласования, Отказ |
| В ремонте | Готов, Ожидание запчастей, Ожидание согласования, Отказ |
| Готов | Выдан, В ремонте (возврат на доработку) |
| Выдан | — (терминальный; повторное обращение = новый заказ со ссылкой на исходный) |
| Отказ | Выдан (возврат устройства без ремонта), Утиль |
| Утиль | — (терминальный) |

\* «Диагностика → Готов» — кейс «неисправность не подтвердилась».

Каждый переход пишется в неизменяемую историю: кто, когда, из какого статуса в какой, комментарий. Откат на произвольный статус — только администратору (тоже с записью в историю).

## 2. Допущения (действуют, пока не оспорены)

| № | Допущение | Обоснование |
|---|---|---|
| A1 | **1 заказ = 1 устройство.** Клиент с тремя устройствами получает три заказа. | Так работают Workpan/RemOnline; упрощает статусы, документы, QR и финансы. |
| A2 | **Одна организация, одна точка приёма.** Мульти-филиальность и multi-tenant — вне MVP. | 1–10 сотрудников; реквизиты организации — в настройках. |
| A3 | **Supabase Cloud** как основной вариант развёртывания; Docker — для фронтенда и опционально self-hosted Supabase. | Минимальный DevOps для команды из 1–10 человек; «развернуть в тот же день». |
| A4 | **Валюта — RUB**, без копеек в UI (хранение в копейках/numeric), часовой пояс — Europe/Moscow, настраивается. | Российский сервисный центр. |
| A5 | **Номер заказа** — сквозной человекочитаемый, формат `<префикс>-<номер>` (например `L-10001`), генерация на стороне БД (sequence), без дыр не гарантируем. | Конкурентная вставка; счётчик в БД — единственный надёжный источник. |
| A6 | **Финансы — только учёт**, без интеграции с онлайн-кассой (54-ФЗ), эквайрингом и бухгалтерией. Статус оплаты: не оплачен / предоплата / оплачен. | Интеграции — отдельный дорогой контур; в MVP фиксируем суммы и факт оплаты. |
| A7 | **Склад запчастей не ведём.** В заказе — только стоимость запчастей (одной суммой или списком строк). | Полноценный складской учёт — фаза 2. |
| A8 | **Гарантия на ремонт** — поле «гарантия, дней» в заказе + дата окончания в гарантийном талоне. Учёт повторных обращений — ссылка «повторный ремонт по заказу №». | Требуется для гарантийного талона из постановки. |
| A9 | **Telegram-уведомления** требуют opt-in клиента: бот не может написать первым по номеру телефона. Клиент подключается по ссылке/QR с токеном заказа. | Ограничение Telegram Bot API. |
| A10 | **WhatsApp в MVP — через провайдера-агрегатора** (абстракция «канал»), включается конфигурацией; по умолчанию активны Telegram + Email. | Официальный WhatsApp Business API платный и требует верификации Meta; неофициальные шлюзы — риск бана номера. |
| A11 | **Email** — через SMTP-совместимого провайдера (настройка в env). | Универсально, без vendor lock-in. |
| A12 | **PWA-офлайн** = офлайн-просмотр закэшированных данных + очередь на повтор неудавшихся запросов. Полноценное офлайн-создание заказов с синхронизацией конфликтов — вне MVP. | Конфликт-резолюшн офлайн-записей — самый дорогой и рискованный кусок PWA. |
| A13 | **Фото** сжимаются на клиенте (canvas/WebWorker, ~до 1920px, JPEG/WebP) перед загрузкой в Storage. | Загрузка с телефона по мобильной сети. |
| A14 | **«Не принят»** = предзаявка (звонок, заявка с сайта, запись): заказ создан, устройство ещё не у нас. | Иначе статус не имеет смысла в потоке. |
| A15 | **Apple ID статус** — это поле-признак для смартфонов Apple: «Локатор/Activation Lock выключен / включён / не проверен». | Стандартная практика приёмки техники Apple. |
| A16 | Публичная QR-страница показывает только: номер заказа, статус, дату приёма, плановую дату готовности, комментарий сервиса, контакты СЦ. Никаких ФИО, телефонов, серийников, сумм. | Прямо из постановки; суммы тоже считаем чувствительными. |
| A17 | Импорт справочников: шаблон CSV/XLSX с колонками `Категория;Бренд;Модель`, режим «добавить недостающее» (upsert), отчёт об ошибках построчно. | Минимально достаточный контракт импорта. |
| A18 | Мастер видит и редактирует **свои** заказы (диагностика, работы, комментарии, статусы ремонтного контура), менеджер — все заказы, админ — всё + справочники, шаблоны полей, пользователи, настройки. | Базовая трактовка RBAC из постановки; уточняется в §3. |

## 3. Уточняющие вопросы

Сгруппированы по влиянию. **На каждый вопрос указан дефолт** — если ответ «по умолчанию» устраивает, достаточно сказать «продолжай», и мы фиксируем дефолты.

### Критичные (влияют на архитектуру и БД)

1. **Заказ и устройства.** Подтверждаете «1 заказ = 1 устройство» (A1)?
   *Дефолт: да.*
2. **Развёртывание.** Supabase Cloud (быстрее, дешевле в поддержке) или полностью self-hosted Supabase в Docker (данные только у вас, но заметно больше DevOps)?
   *Дефолт: Cloud, с docker-compose для self-hosted как документированной опцией.*
3. **Точки приёма.** Одна точка/одно юрлицо (A2)? Нужен ли задел под филиалы в модели данных (поле `branch_id` с первого дня)?
   *Дефолт: одна точка, без задела — добавление филиалов потом потребует миграции, но не перестройки.*
4. **Финансы.** Достаточно учёта сумм и статуса оплаты без кассы/эквайринга (A6)? Нужен ли учёт способа оплаты (нал/безнал/перевод)?
   *Дефолт: да, плюс поле «способ оплаты».*

### Важные (влияют на объём MVP)

5. **WhatsApp.** Есть ли уже аккаунт WhatsApp Business API / провайдер (Twilio, 360dialog, Wazzup и т.п.)? Если нет — согласны, что в MVP реально работают Telegram + Email, а WhatsApp подключается конфигурацией позже (A10)?
   *Дефолт: Telegram + Email в MVP.*
6. **Прайс-лист работ.** Нужен ли справочник типовых работ с ценами (выбор из списка при оформлении) или стоимость вводится вручную?
   *Дефолт: вручную в MVP, прайс-лист — фаза 2.*
7. **Запчасти в заказе.** Одной суммой или списком строк (наименование × цена × кол-во)? Список нужен для корректного акта выполненных работ.
   *Дефолт: список строк (без привязки к складу).*
8. **SMS.** Точно не нужен как канал уведомлений?
   *Дефолт: не нужен.*

### Уточняющие (влияют на детали)

9. **Формат номера заказа** (A5): какой префикс? Нумерация сквозная или с начала года?
   *Дефолт: сквозная, префикс настраивается в настройках организации.*
10. **Реквизиты для PDF:** ИП или ООО, нужны ли поля ИНН/ОГРН, подпись/печать на квитанции? (Шаблоны сделаем настраиваемыми, но базовый макет должен соответствовать вашей практике.)
    *Дефолт: настраиваемые реквизиты организации: название, ИНН, адрес, телефон, режим работы; места под подписи сторон.*
11. **Удаление пользователей/клиентов:** при увольнении сотрудника — деактивация аккаунта (история сохраняется)?
    *Дефолт: да, только деактивация.*
12. **Язык интерфейса** — только русский, без i18n-каркаса?
    *Дефолт: только русский (тексты всё равно выносим в константы).*

## 4. Границы MVP vs развитие

### MVP (этапы 2–18 текущего плана)

- Заказы: полный жизненный цикл, state machine, история статусов.
- Клиенты, устройства, динамические справочники (Категория → Бренд → Модель) с импортом CSV/XLSX, серверным поиском и автодополнением (десятки тысяч моделей).
- Динамические поля по категориям (шаблоны админа + JSONB-значения, без миграций).
- Финансы заказа: работы, запчасти (строками), предоплата, итог, статус и способ оплаты.
- RBAC (админ/менеджер/мастер) + RLS на сервере.
- Файлы: фото устройства/серийника, документы; сжатие на клиенте; Supabase Storage.
- QR-код + публичная страница статуса по неугадываемому токену.
- PDF: квитанция, акт выполненных работ, акт выдачи, гарантийный талон.
- Уведомления: Telegram + Email (идемпотентные, с outbox-очередью); WhatsApp — через подключаемый провайдер.
- Поиск (номер, телефон, имя, серийник, бренд, модель) и фильтры (статус, мастер, менеджер, дата, категория, бренд).
- Дашборд: таблица заказов + 6 виджетов из постановки.
- Аудит: неизменяемый журнал изменений, soft delete повсюду.
- PWA: установка, кэш оболочки и справочников, очередь повторов.
- Docker, тесты, инструкция деплоя, финальный аудит.

### Фаза 2 (сознательно за рамками MVP)

- Склад и закупки запчастей; прайс-лист работ.
- WhatsApp как полноценно сконфигурированный канал (когда будет провайдер); SMS.
- Аналитика/отчёты (выручка по мастерам, средний чек, конверсия отказов).
- Мульти-филиальность, multi-tenant.
- Интеграция с онлайн-кассой (54-ФЗ), эквайринг, экспорт в бухгалтерию.
- Личный кабинет клиента (история всех его ремонтов), онлайн-запись.
- Полноценное офлайн-создание заказов с синхронизацией.
- Зарплаты/мотивация мастеров (% от работ).

## 5. Риски

| Риск | Влияние | Митигация |
|---|---|---|
| **WhatsApp:** официальный API платный/долгая верификация; «серые» шлюзы банят номера | Канал может не заработать к запуску | Абстракция «канал уведомлений»; запуск на Telegram + Email; WhatsApp — конфигурацией |
| **Telegram opt-in:** бот не может написать клиенту первым | Часть клиентов не получит уведомления | Ссылка/QR на бота печатается в квитанции; Email как fallback |
| **Динамические поля:** валидация JSONB по шаблону — точка сложности | Грязные данные, расхождение шаблона и значений | Валидация на сервере (триггер/функция) + на клиенте; версионирование шаблонов, старые значения не ломаются |
| **RLS-политики:** сложные правила → ошибки доступа или деградация | Утечка данных или медленные запросы | Минимум хитрых политик, покрытие тестами на каждую роль, EXPLAIN на ключевых запросах |
| **Импорт справочников:** грязные данные, дубли, кодировки | Замусоренный справочник | Нормализация (trim/регистр), upsert по уникальному ключу, dry-run с отчётом ошибок |
| **Поиск по 10⁴–10⁵ моделей** | Деградация автодополнения | pg_trgm + GIN-индексы, серверная пагинация, debounce на клиенте |
| **PWA-офлайн:** соблазн «полного офлайна» | Конфликты данных, срыв сроков | Жёстко ограничить scope: кэш чтения + очередь повторов (A12) |
| **Supabase lock-in** | Сложность миграции | Это стандартный PostgreSQL + S3-совместимый Storage; бизнес-логика в SQL/Edge Functions переносима |
| **Публичная QR-страница** | Перебор токенов → утечка статусов | Токен ≥ 128 бит энтропии, rate limiting, никаких ПД на странице (A16) |
| **Один «большой» этап = один большой артефакт** (например, полный DDL) | Ошибки из-за объёма | Стоп-гейты по этапам, ревью пользователем, тесты на критичную логику (state machine, RLS) |

## 6. Что будет на следующем этапе

**Этап 2 — Проектирование архитектуры:** компонентная схема (SPA ↔ Supabase: PostgREST, Auth, Storage, Edge Functions), карта сервисов, потоки данных ключевых сценариев (создание заказа, смена статуса → уведомление, публичная QR-страница, импорт справочника), ключевые архитектурные решения с обоснованием (включая стратегию динамических полей и схему уведомлений через outbox).
