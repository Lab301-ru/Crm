## 2. Что именно проверяет logic_test (все — OK)

1. **Нумерация**: номера последовательны, display_number с префиксом организации.
2. **Машина статусов**: переход вне таблицы переходов отклонён (менеджер), статус не изменился.
3. **Единственная дверь**: прямой `UPDATE orders.status` мимо `change_status()` запрещён триггером.
4. **Коррекция админом**: админ может переход вне таблицы (ручное исправление ошибок).
5. **Идемпотентность outbox**: повторное «готов» (ready → in_repair → ready) не дублирует уведомления (`event_key`).
6. **Закрытый заказ**: вставка работ в выданный заказ заблокирована для менеджера.
7. **Доп-поля**: неизвестный ключ — отказ; число строкой — отказ; корректные значения — приняты.
8. **Шаблон поля**: `key`/тип неизменяемы (целостность старых данных), label менять можно.
9. **Телефон**: `8 (917) 000-11-22` нормализуется в `+79170001122`.
10. **Soft delete**: физический DELETE заказа запрещён; история статусов append-only.
11. **quick_add_model**: идемпотентен без учёта регистра (нет дублей справочника).
12. **Импорт каталога**: вставка/пропуск дублей/построчные ошибки; повторный импорт ничего не дублирует.
13. **Глобальный поиск**: находит заказ по фрагменту телефона и по серийному номеру.

## 3. Тесты Edge Functions

`template.test.ts` — рендер шаблонов уведомлений (подстановка плейсхолдеров, tracking_url из qr_token, «уточняется» для пустой/кривой даты, дефолтный шаблон, незнакомый плейсхолдер не роняет рендер). **Без внешних зависимостей** — собственный assert, работает офлайн и в CI без доступа к jsr.io.

Логика доставки (notify-dispatch) сетевая по своей природе — её покрытие на стороне БД: идемпотентный claim (`claim_notifications`) и backoff проверяются через outbox-проверки rls/logic-тестов; живой smoke — на Этапе 17 при деплое.

## 4. Как запускать

```bash
npm run test:db          # поднятый PostgreSQL: миграции + seed + 28 проверок + демо
npm run test:functions   # deno test (5 проверок)
npm run typecheck && npm run build
```

`test-db.sh` принимает host/port (`./scripts/test-db.sh /tmp/pgtest 5544`), создаёт БД с нуля и применяет всё из репозитория — никакого подготовленного состояния.

## 5. Риски и открытые вопросы

- E2E браузерных тестов (Playwright) нет сознательно: для команды 1–10 человек поддержка E2E дороже пользы; ядро (роли, машина статусов, валидации) покрыто на уровне БД, где живёт вся логика.
- Проверки в тестах ловят `exception when others` с фильтром своих `FAIL:` — этого достаточно: важен факт отказа и неизменность данных, а не текст ошибки.
